# Chloe × AI 产品原型落地方法论

> 从产品想法到 GitHub 上线的完整协作手册
> 沉淀自：StageShare / 微信读书AI / JoyOps / Portfolio 四个项目

---

## 一、启动：把模糊想法变成清晰方向

**你需要告诉 AI 的四件事：**

1. **用户是谁** — 谁在用这个产品？他们有什么特征？
2. **核心痛点** — 他们现在是怎么解决这个问题的？哪里不够好？
3. **你的方案** — 你想怎么解决？哪个功能是最核心的一个？
4. **差异点** — 为什么是你，而不是已有的产品？

**AI 会做的事：** 追问补全 → 用一段话总结方向 → 你确认后再继续

> ⚠️ 不要跳过确认步骤，方向错了越往后返工成本越高

## 二、PRD：结构化你的想法

**标准 PRD 结构：**
```
产品概述（一句话定位）
用户画像
核心用户旅程（主路径 1-2 条）
功能列表（按优先级：P0/P1/P2）
设计约束（平台/技术限制）
成功指标
```

**关键原则：**
- 始终以 **2条主路径** 为骨架：生产者视角 + 消费者视角
- 功能列表要有优先级，不是所有想法都要第一版做
- 写清楚「不做什么」和「为什么不做」

**功能优先级判断框架：**

| 维度 | 问题 |
|---|---|
| 阻断性 | 没有它，核心路径能走通吗？ |
| 用户频次 | 这个功能用户多久用一次？ |
| 差异性 | 这是你和竞品最不同的地方吗？ |
| 实现成本 | 做起来有多复杂？ |

**P0**：阻断核心路径 → 必须有
**P1**：高频 + 有差异性 → 第一版做
**P2**：低频或成本高 → 迭代再加

## 三、需求描述：让 AI 第一次就做对

### 开新项目时给 AI 一份「设计简报」

```
风格参照：[网站/截图链接]
配色：[主色/辅色的具体色值]
字体偏好：[衬线/无衬线/具体字体名]
不要出现：[xxx]
设备：[移动端/PC端]
这个页面要完成的一件事：[xxx]
```

> 一份简报，省掉后面 80% 的来回。

### 给「黄金参照物」，不要描述感觉

> ✅ "参考 aristidebenoist.com 的画廊风格，左大图右缩略图"
> ❌ "做一个好看的图片展示区"

每次开新项目，第一件事：找 2-3 个你觉得审美对的网站/截图发给 AI。

### 用「不要做什么」限定边界

AI 最容易在你没说的地方乱发挥。明确说不要什么，比说要什么更有效。

> "不要全屏弹窗，不要动画太复杂，不要把标签贴在图片上"

### 描述页面时说「层级」不说「感觉」

| 模糊 ❌ | 清晰 ✅ |
|---|---|
| "整体简洁一点" | "去掉副标题，只保留大标题和一行说明" |
| "留白多一点" | "卡片之间 gap 加大，内边距从 1rem 改 2rem" |
| "字小一点" | "标题从 3.8rem 改 2.8rem，说明文字 0.85rem" |

### 描述一个页面的标准格式

```
【页面名称】
这个页面的目标：用户来这里是为了完成什么？

【核心元素】（从上到下列出）
- 顶部：xxx
- 主体：xxx（列表/卡片/表单/图文）
- 底部/固定：xxx

【关键交互】
- 点击 A → 发生什么
- 滑动/长按 → 发生什么
- 为空时 → 显示什么
- 加载中 → 显示什么

【来源与去向】
- 从哪里进入这个页面
- 可以跳转去哪里
```

## 四、功能设计：补全自己容易漏掉的地方

每次设计完功能，对照这四个维度检查：

**1. 异常路径**
- 用户做了「错误」操作怎么办？（误触、输错、反悔）
- 网络断了/加载失败怎么办？
- 0条数据时显示什么？第一次使用有没有引导？

**2. 双边/多角色视角**
- 每个角色的核心路径各是什么？
- A 做了某个操作，B 那边会发生什么？

**3. 时间维度**
- 第一次用 vs 用了很久的用户，体验一样吗？
- 用户离开再回来，状态保留了吗？
- 有没有「等待」环节？等待时用户看什么？

**4. 激励与动机**
- 用户完成操作后，有没有正向反馈？
- 什么驱动用户回来？
- 冷启动期没有内容/没有人，用户还会留下来吗？

> 💡 随时可以问 AI："帮我检查这个功能有没有遗漏"

## 五、原型迭代：高效改稿不翻车

### 正确节奏

```
① AI 生成第一版 → Canvas/预览链接确认
② 截图反馈（直接上传截图，说清楚哪里有问题）
③ 每次只改一件事，改完预览确认再继续
④ 满意后推 GitHub Pages 上线
```

### 反馈时定位到具体元素

> ✅ "三扇门的大字和下面中文描述重叠了"
> ❌ "字体重合了"

**截图 + 圈出问题位置 = 最高效的反馈方式**

### 避坑清单

- **不要** 一次提多个修改需求（容易顾此失彼）
- **不要** 跳过预览直接推 GitHub
- **不要** 在对话里口述关键限制（AI 上下文有限，写进文件/注释）
- **改崩了** → 从 GitHub 拉干净版本重来，不在坏版本上修补

## 六、上线：部署到 GitHub Pages

**标准步骤：**
```
① 本地预览确认没问题
② AI 用 GitHub API 推送
③ 等待 1-2 分钟 GitHub Pages 生效
④ 打开线上链接最终确认
```

**GitHub Token 获取（每次需要时操作）：**
> Settings → Developer settings → Personal access tokens → **Tokens (classic)**
> 勾选 `repo` 全部权限，有效期 7 天够用
> ⚠️ 必须是 Classic Token，Fine-grained Token 权限受限会报 403

## 七、收尾：面试弹药提炼

每个项目完成后，沉淀三样东西：

**① 三大设计亮点**（"我做了一个特别有意思的决策"）
- 要具体，要有用户视角的原因
- 示例：StageShare「现场交换取货」→ 因为随舞用户本来就在同一个现场，规避了物流等待

**② AI 协作复盘**（体现 PM 思维的加分项）
- 遇到了什么问题？怎么解决的？
- 证明你能驾驭 AI 工具，不是只会发指令

**③ 产品能力体现**
- 需求拆解：怎么从模糊想法到清晰功能？
- 优先级判断：为什么先做这个不做那个？
- 用户视角：设计决策的出发点是什么？
